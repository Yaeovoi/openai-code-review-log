1.  **性能优化（减少 API 调用）**
    *   **变更**：将版本获取逻辑统一收敛到 `action.yml` 的 `get-version` 步骤，通过环境变量 `JAR_VERSION` 传递给脚本。

2.  **安全性增强（强制校验）**
    *   **变更**：在 `run-review.sh` 中，将 SHA256 校验从“警告并跳过”改为“错误并退出”。

    *   环境变量分组增加了注释，提高了可读性。

### ⚠️ 潜在问题与建议

**1. 异常处理逻辑的边缘情况**
*   **问题描述**：
    当 `action.yml` 中获取版本失败时，`VERSION` 被设为 `unknown`。
    在 `run-review.sh` 中，逻辑如下：
    ```bash
    local latest_tag="${JAR_VERSION:-unknown}"
    # ...
    if [ "${latest_tag}" = "unknown" ]; then
      echo "警告：无法确定最新版本，使用缓存"
      return 0
    fi
    ```
    如果此时缓存不存在，`need_download` 仍为 `true`，脚本会继续执行下载逻辑。下载完成后，脚本末尾（虽然 diff 未显示完全，推测存在）会将 `${latest_tag}` 写入 `${VERSION_FILE}`。
    这意味着 `${VERSION_FILE}` 的内容将变成字符串 "unknown"。
*   **后果**：如果某次运行因网络问题获取版本失败，缓存被标记为 "unknown"。后续即使网络恢复，如果 API 依然返回 "unknown"（虽然概率低），或者后续逻辑比对时，"unknown" != "v1.0.0"，会触发不必要的更新。更严重的是，如果远端有更新，但本地一直记录为 "unknown"，可能会导致版本比对逻辑混乱。
*   **建议**：
    在保存版本号到文件时，增加判断，避免将 "unknown" 写入缓存文件：
    ```bash
    if [ "${latest_tag}" != "unknown" ]; then
        echo "${latest_tag}" > "${VERSION_FILE}"
    else
        # 可选：删除旧的版本文件，强制下次重试
        rm -f "${VERSION_FILE}"
    fi
    ```

**2. 文件末尾换行符**
*   **观察**：`action.yml` 的 diff 显示 `No newline at end of file`。
*   **建议**：标准文本文件应以换行符结尾，建议在文件末尾添加一个空行，以符合 POSIX 标准，避免某些旧版工具处理时出现警告。

**3. Shell 变量引用健壮性**
*   **观察**：在 `download_and_verify_jar` 函数开头：
    ```bash
    local latest_tag="${JAR_VERSION:-unknown}"
    ```
*   **建议**：这是一个好的开始。请确保后续所有使用 `latest_tag` 的地方都使用了双引号（如 `"${latest_tag}"`），防止变量展开时的词法拆分，从 diff 看这部分做得是好的，请保持。
